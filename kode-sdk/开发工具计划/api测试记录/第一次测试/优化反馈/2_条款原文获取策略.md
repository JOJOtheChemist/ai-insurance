# 反馈 2：条款原文获取策略完善

**来源场景**: 场景一 (30岁程序员)

**问题描述**:
用户敏锐指出，目前的搜索结果（`extend_info`）仅展示了类似 `"sudden_death": "猝死保险金"` 的结构化标签。
这只是“亮点展示”，缺乏法律效力。Agent 在向用户解释条款时，应该基于**完整的法律条款原文**，而不能只看标签发挥。

**现场核实**:
经数据库核查 (Product ID 40)：
*   `extend_info`: 仅存储 Tags (e.g. "险种类型: 意外险")。
*   `coverage`: 存储了完整的条款列表 (JSON String)，例如：`"• 猝死保险金：在保险合同有效期内，若被保险人发生猝死..."`。目前 Agent 未调用此字段。

**优化建议**:
1.  **Prompt 调整**: 明确指示 Agent，当用户询问具体保障细节时，必须在 `inspect` 工具中请求 `coverage` 字段。
2.  **后端优化**: 考虑在 `inspect` 接口增加对 `coverage` 字段的自动美化处理。
    *   *现状 (Raw JSON Array of Strings)*: 
        ```json
        [
          "• 意外身故保险金：若被保险人身故...", 
          "• 猝死保险金：若被保险人发生猝死..."
        ]
        ```
        (目前是一个“字符串列表”，每一项包含标题和内容，混在一段长文本里)
    *   *优化方案 A (Plain Text)*: 简单的拼接，适合通读。
    *   *优化方案 B (Key-Value 结构化)* **[已选定]**: 
        后端尝试正则解析，将其转为更结构化的字典，例如：
        ```json
        {
          "意外身故保险金": "若被保险人身故...",
          "猝死保险金": "若被保险人发生猝死..."
        }
        ```
        这样 AI 更容易按 Key 索骥，精准提取。这是对数据的进一步精洗，价值更高。

**决策结论**:
*   [x] **采用方案 B**：在 Backend 增加解析逻辑，将 `coverage` 清洗为 Key-Value 格式返回。
